* Sprint Overview

The cprr tool is functional with 19 confirmed conjectures and a verified state
machine (Alloy). The next sprint shifts from "does the tool work?" to "does the
methodology scale, compose, and integrate?"

Three themes emerge from the open issues and untested hypotheses:

1. *Composition* — Can multiple agents/humans use cprr concurrently without conflicts?
2. *Verification* — Can we extend formal guarantees beyond the state machine?
3. *Adoption* — Can we reduce friction for new users and AI agents?

** Tier 1 — Unblock Everything (do first, in parallel if possible)

*** 004: Refinery Pattern for Merge Conflicts
:PROPERTIES:
:BEAD: CPRR-lqz
:PRIORITY: P1
:EFFORT: M (2–3 days)
:IMPACT: High — unlocks multi-agent
:END:

**** Conjecture

Concurrent writes to =conjectures.json= can be resolved without data loss using
a last-writer-wins CRDT with logical timestamps per conjecture ID.

**** Hypothesis

If each conjecture entry carries a Lamport timestamp and updates are merged by
max-timestamp-per-ID, then two agents modifying disjoint conjectures
simultaneously will produce a correct merged result in 100% of test cases.

**** Falsification Criteria

- Any test case where a conjecture written by Agent A is absent from the merged
  result after Agent B's concurrent write
- Any test case where evidence ordering within a single conjecture is corrupted
- Merge operation taking >100ms on a file with 1000 conjectures

**** Method

1. Add =updated_at= (Unix nanos) field to conjecture JSON
2. Implement =cprr merge <path>= that does per-ID max-timestamp resolution
3. Simulate concurrent writes with two goroutines + file locking removed
4. Measure data integrity and latency

*** 005: Bead Semantics for Isolation
:PROPERTIES:
:BEAD: CPRR-a58
:PRIORITY: P1
:EFFORT: S (1–2 days)
:IMPACT: High — defines the mental model
:END:

**** Conjecture

Git worktrees provide sufficient isolation for parallel hypothesis testing if
each worktree operates on its own =.cprr/= snapshot and merges via experiment 004.

**** Hypothesis

A developer can run =cprr add= in two different worktrees simultaneously, and
after pushing both branches and merging, the main branch's =.cprr/conjectures.json=
will contain both conjectures with no manual conflict resolution.

**** Falsification Criteria

- Git merge conflict in =conjectures.json= requiring manual intervention
- Lost conjecture after merge
- Duplicate conjecture IDs post-merge

**** Method

1. Script: create two worktrees, add one conjecture in each, push, merge
2. Run 10 iterations, check for any failure
3. If conflicts occur, apply experiment 004's refinery merge as a git merge driver

*** 008: Property-Based Testing for Guards
:PROPERTIES:
:PRIORITY: P1
:EFFORT: S (1 day)
:IMPACT: Medium — strengthens core, fast to do
:END:

**** Conjecture

The guard system's invariants (no confirmation without 2+ evidence, no refutation
without 1+ evidence) hold for all reachable states, not just the hand-written
test cases.

**** Hypothesis

Using Go's =rapid= library to generate random sequences of state transitions, the
guard system will reject all invalid transitions and accept all valid ones across
10,000 generated test cases.

**** Falsification Criteria

- Any generated sequence that reaches =confirmed= with <2 evidence entries
- Any generated sequence that reaches =refuted= with 0 evidence entries
- Any sequence where =abandoned= is unreachable

**** Method

** Tier 2 — Extend Guarantees and DX

*** 006: Experiment Number Deconfliction
:PROPERTIES:
:BEAD: CPRR-b6t
:PRIORITY: P2
:EFFORT: S (1 day)
:IMPACT: Medium — prevents annoying collisions
:END:

**** Conjecture

Branch-based experiment numbering (allocating ranges or using branch-prefixed
IDs) eliminates duplicate experiment numbers when multiple developers create
experiments concurrently.

**** Hypothesis

If experiment directories use the format =NNN-<branch-short>-name/= (e.g.,
=004-feat-refinery-pattern/=), then two developers creating experiments on
different branches will never collide, and the sort order remains meaningful.

**** Falsification Criteria

- Two experiments with identical directory names created on different branches
- Lexicographic sort of experiment dirs not reflecting creation order within a branch

*** 009: Alloy Liveness Extensions
:PROPERTIES:
:PRIORITY: P2
:EFFORT: M (2 days)
:IMPACT: Medium — completes formal story
:END:

**** Conjecture

The existing Alloy model can be extended to verify liveness properties: every
conjecture can eventually reach a terminal state (confirmed, refuted, or
abandoned), and no conjecture can be trapped in a non-terminal state.

**** Hypothesis

Adding temporal assertions to =cprr.als= will verify that for all reachable
states, at least one transition to a terminal state exists. The Alloy Analyzer
will find no counterexamples within scope 6.

**** Falsification Criteria

- Alloy finds a counterexample: a reachable state with no outgoing transitions
  to a terminal state
- The model requires scope >6 to find violations (suggests the property is
  vacuously true at small scope)

*** 011: SKILL.md Agent Packaging
:PROPERTIES:
:PRIORITY: P2
:EFFORT: S (1–2 days)
:IMPACT: High — enables AI agent adoption
:END:

**** Conjecture

An AI coding agent (Claude Code, Cursor, etc.) given a SKILL.md file describing
cprr's workflow can autonomously create, test, and advance conjectures without
human intervention for the mechanical steps.

**** Hypothesis

Given SKILL.md + AGENTS.md, an AI agent tasked with "investigate whether cprr
supports tags on evidence entries" will: (1) create a conjecture, (2) write a
test, (3) add evidence, and (4) advance the state correctly — all via CLI
commands.

**** Falsification Criteria

- Agent invokes a command that doesn't exist
- Agent skips the hypothesis step (violates guards)
- Agent marks confirmed without adding sufficient evidence
- Agent cannot recover from a guard rejection

** Tier 3 — Integration and Capstone

*** 007: Multi-Agent Smoke Test
:PROPERTIES:
:PRIORITY: P3
:EFFORT: M (2–3 days)
:IMPACT: Very High — validates entire composition theme
:DEPENDS: 004, 005, 006, 011
:END:

**** Conjecture

Two AI agents working on the same cprr repository via separate worktrees can
each independently create, test, and resolve conjectures, with all work
correctly merged into main.

**** Hypothesis

Running two Claude Code instances in parallel worktrees, each given a different
hypothesis to investigate, will result in a merged main branch containing both
conjectures in their correct final states with complete evidence chains.

**** Falsification Criteria

- Any data loss in merged =conjectures.json=
- Any conjecture in an inconsistent state post-merge
- Either agent blocked or deadlocked waiting for the other

*** 010: CI Guard Verification
:PROPERTIES:
:PRIORITY: P3
:EFFORT: S (1 day)
:DEPENDS: 008, 009
:END:

**** Conjecture

A CI pipeline can enforce that no PR merges if it contains a conjecture
transition that violates guards, providing the same safety guarantees at the
repository level that the CLI provides locally.

**** Hypothesis

A GitHub Actions workflow running =cprr doctor= + the property-based test suite
will catch a deliberately invalid state transition in a test PR and fail the
build.

**** Falsification Criteria

- CI passes a PR containing a =confirmed= conjecture with 0 evidence entries
- CI takes >60s to run the full verification suite

*** 012: Real-Project Dogfood
:PROPERTIES:
:PRIORITY: P3
:EFFORT: L (1 week, ongoing)
:DEPENDS: 003, 011
:END:

**** Conjecture

Applying cprr to an existing project (e.g., SEFACA or JITIR) will surface at
least 3 assumptions that would otherwise go untested, and the refuted hypotheses
will save more development time than the overhead of maintaining conjectures.

**** Hypothesis

Over one week of using cprr on a real project, at least 1 in 5 conjectures will
be refuted, and the developer (JW) will rate the methodology as net-positive for
decision quality on a 1–5 scale.

**** Falsification Criteria

- 0 refutations in 10+ conjectures (suggests rubber-stamping, not testing)
- Developer rates methodology as net-negative (≤2/5) for overhead vs. insight
- Conjectures become stale (>5 days in =open= without advancing)

*** 013: Cross-Repo Conjecture Sync
:PROPERTIES:
:PRIORITY: P3
:EFFORT: L (3–5 days)
:DEPENDS: 007, 010, 012
:END:

**** Conjecture

A =cprr sync= command can aggregate conjectures from multiple repositories into
a unified view, enabling team-wide hypothesis dashboards without centralized
storage.

**** Hypothesis

Using =ghq list= to enumerate repositories containing =.cprr/=, a sync command
can produce a merged JSON view of all conjectures across repos in <5 seconds for
JW's 500+ repository collection.

**** Falsification Criteria

- Sync takes >5 seconds for 500 repos (even if most lack =.cprr/=)
- Conjecture ID collisions across repos without namespacing
- Dashboard data goes stale within 1 hour of a local update
